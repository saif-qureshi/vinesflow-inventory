# Vineflow ‚Äî Sales, Purchases, FBR e-Invoicing & Accounting Plan

Status: **design / approved-pending**. No code written yet.
Scope: the transactional document layer (invoices, orders, bills, payments,
returns), the conditional **FBR Digital Invoicing** integration it must carry
from day one, and the seam where the **accounting ledger** clips on later.

---

## 0. Principles

1. **One shared document engine**, not seven features. Every sales/purchase
   document is a row in `documents` + `document_lines`, discriminated by `type`.
2. **FBR e-invoicing is a conditional layer, built last.** It is *off* unless an
   org opts in, and its fields + integration land in the **final phase (¬ß6)**.
   Deferring is safe: the FBR columns are just another autogenerated migration
   when we reach them.
3. **Documents are the bridge.** A finalized document drives (a) the stock
   ledger, (b) ‚Äî later ‚Äî the accounting GL, (c) ‚Äî conditionally ‚Äî an FBR filing.
   None of those three know about each other; each keys back to the document.
4. **Accounting-ready by construction.** Everything built now ‚Äî documents, lines,
   payments, stock movements ‚Äî preserves the full financial detail and the
   `(source_type, source_id)` back-reference the future GL needs, so **Books can
   post *and backfill* from existing data with zero rework**. The GL is never a
   reason to reshape what we build now; it only reads it. See ¬ß5 for the hooks.
5. Conventions unchanged: SQLAlchemy 2.0 sync, service layer + DI, EnvelopeRoute,
   cursor pagination, **schema via autogenerated Alembic migrations from the
   models** (`alembic revision --autogenerate` ‚Üí review ‚Üí `upgrade head`), seed
   master data on org creation, per-module RBAC. Currency **PKR**. antd + theme
   tokens only.

## 1. FBR e-invoicing requirements (built last, ¬ß6)

FBR (Pakistan's tax authority) mandates registered persons integrate their
ERP/POS with the **Digital Invoicing (DI) API** via PRAL. On finalizing a sale
you POST the invoice; FBR returns an **IRN** (Invoice Reference Number) + **QR**
that must be printed. This is deferred to the final phase ‚Äî captured here so the
core engine (Phases 1‚Äì5) is built without blocking on it. The DI payload demands
fields the FBR phase will add:

- **Per item (13 mandatory):** HS Code, UoM (FBR code list), Sale Type,
  SRO Schedule No, SRO Item Serial No, rate, value excl. sales tax, sales tax,
  further tax, extra tax, FED payable, fixed/retail price, discount.
- **Per party (buyer):** NTN/CNIC (have), **province**, **registration type**
  (Registered / Unregistered ‚Äî required in the FBR payload as `buyerRegistrationType`).
- **Per org (seller):** NTN (have), **province**, business name, address (have),
  **IRIS security token** (sandbox vs production routing is by token).

So products carry tax classification, lines carry a tax breakdown, and
org/party carry province + registration. These exist whether or not FBR is on ‚Äî
FBR just consumes them.

### 2.3 `payments` + `payment_allocations`

A payment (received/made) can settle several invoices/bills.

- `payments`: `id, org_id, type(received|made), party_id, number, date, amount, method, reference, notes, +audit`
- `payment_allocations`: `id, payment_id, document_id, amount` ‚Üí drives
  `documents.amount_paid` / `balance_due` and the Paid/Partial status.

### 2.4 `tax_rates`, `document_sequences`

- `tax_rates`: org-level registry ‚Äî `name ("GST 18%"), rate, is_active`. Seeded
  with **18%** (current standard) + **0% / Exempt**.
- `document_sequences`: `org_id, type, prefix, next_number, padding` ‚Äî row-locked
  for safe concurrent numbering.

### 2.5 New/extended fields on existing tables

> All of ¬ß2.5 lands in the **FBR phase (¬ß6)**, not up front. Core invoicing
> (Phases 1‚Äì5) uses a single line tax rate (`document_lines.tax_rate_id`); the
> FBR-mandated **per-item tax rate** (driven by HS code / sale type / SRO) is
> part of the FBR phase.

**`products`** (leaf items ‚Äî the FBR/tax classification):
`hs_code`, `uom_code` (FBR UoM), `sale_type` (FBR sale-type / tax category),
`sro_schedule_no`, `sro_item_serial_no`, `default_tax_rate_id`.

**`parties`** (buyer identity for FBR): add `province`, `registration_type`
(`registered` | `unregistered` ‚Äî reported as `buyerRegistrationType`, no tax
logic attached). NTN/STRN/CNIC already exist.

**`organizations`** (seller + FBR credentials): add `province`, and an
**FBR settings** group ‚Äî `fbr_enabled`, `fbr_environment` (sandbox|production),
`fbr_token` (encrypted), `fbr_business_name`. (Token stored encrypted at rest.)

### 2.6 FBR reference-data cache (`fbr_*` lookup tables)

FBR publishes lookup lists we must validate against and let users pick from.
Cache them per environment, refreshed from FBR's `pdi/v1` reference endpoints:
`fbr_provinces`, `fbr_uom`, `fbr_hs_codes`, `fbr_sale_types`,
`fbr_doc_types`, `fbr_sro_schedules`, `fbr_sro_items`, `fbr_sale_type_rates`.

## 3. FBR Digital Invoicing integration

### 3.1 Endpoints (PRAL DI API v1.x)

| purpose | URL |
|---|---|
| Validate (dry-run) | `‚Ä¶/di_data/v1/di/validateinvoicedata` (`‚Ä¶_sb` sandbox) |
| Post (file) | `‚Ä¶/di_data/v1/di/postinvoicedata` (`‚Ä¶_sb` sandbox) |
| Reference lookups | `‚Ä¶/pdi/v1/{provinces,uom,itemdesccode,transtypecode,doctypecode,SroSchedule,sroitemcode,SaleTypeToRate}` |

Auth: `Authorization: Bearer <IRIS token>`. **Sandbox vs production is chosen by
which token is set** ‚Äî same code path.

### 3.2 Flow (on invoice finalize, only if `org.fbr_enabled`)

```
finalize invoice
  ‚Üí build FBR payload (seller from org, buyer from party, items from lines)
  ‚Üí POST validateinvoicedata      (catch errors early)
  ‚Üí POST postinvoicedata
     ‚îú‚îÄ success ‚Üí store fbr_irn, fbr_qr, fbr_status=posted, fbr_posted_at
     ‚îî‚îÄ failure ‚Üí fbr_status=failed, fbr_error=<response>   (retryable)
```

Key rules:
- **Conditional & non-blocking.** If FBR is off, `fbr_status=not_applicable` and
  the invoice is fully usable. If FBR is on and the call fails, the invoice still
  exists operationally; we surface the error and allow **Retry**. FBR failure
  never loses the sale.
- **QR/IRN on the PDF.** Once posted, the invoice print shows IRN + QR.
- **Debit/Credit notes** send `fbr_invoice_type="Debit Note"` +
  `fbr_invoice_ref_no` = the original IRN.
- **Further tax** is a manual passthrough field only (default 0); we do **not**
  auto-compute it from buyer registration.
- **Go-live gate:** FBR requires passing the ~28 sandbox scenarios (SN001‚ÄìSN028)
  before issuing a production token. We expose a sandbox "scenario test" mode.

### 3.3 Payload mapping (reference)

Header ‚Üê org (`sellerNTNCNIC, sellerBusinessName, sellerProvince, sellerAddress`),
party (`buyerNTNCNIC, buyerBusinessName, buyerProvince, buyerAddress,
buyerRegistrationType`), document (`invoiceType, invoiceDate, invoiceRefNo,
scenarioId`).
Items ‚Üê each line: `hsCode, productDescription, rate, uoM, quantity,
valueSalesExcludingST, salesTaxApplicable, salesTaxWithheldAtSource, furtherTax,
extraTax, fedPayable, fixedNotifiedValueOrRetailPrice, discount, saleType,
sroScheduleNo, sroItemSerialNo, totalValues`.

## 4. Document ‚Üí stock / money / FBR map

| Doc | Dir | Stock | Money (‚Üí GL later) | FBR |
|---|---|---|---|---|
| Sales Order | sales | commits (no move) | ‚Äî | ‚Äî |
| Delivery Challan | sales | **OUT** on dispatch | ‚Äî | ‚Äî |
| **Invoice** | sales | **OUT** on finalize *(skipped if shipped via challan)* | AR‚Üë, Revenue‚Üë, Tax‚Üë | **posts (conditional)** |
| Sales Receipt | sales | **OUT** | cash sale = invoice+payment | posts (conditional) |
| Credit Note / Return | sales | **IN** | AR‚Üì | Debit/Credit note (conditional) |
| Purchase Order | purchase | incoming (no move) | ‚Äî | ‚Äî |
| Goods Receipt | purchase | **IN** on receive | ‚Äî | ‚Äî |
| Bill | purchase | **IN** on finalize *(skipped if received via GRN)* | AP‚Üë, Inventory/Expense‚Üë | ‚Äî (purchase side later) |
| Vendor Credit | purchase | **OUT** | AP‚Üì | ‚Äî |
| Payment (recd/made) | ‚Äî | ‚Äî | Bank ‚Üî AR/AP | ‚Äî |

**Stock leaves/enters once, on the dispatch/receipt document.** Reuses the
existing ledger: whichever doc represents the physical move posts to
`stock_movements` with `reference_type`/`reference_id`. If a Delivery Challan
already shipped the goods, the Invoice generated from it bills only (no second
movement); same for GRN ‚Üí Bill. When there's no challan/GRN, the Invoice/Bill
carries the movement itself. SO/PO never move stock ‚Äî their open lines populate
the inventory **committed / to-be-received** columns we stubbed.

State machines: Invoice `Draft‚ÜíSent‚ÜíPartially Paid‚ÜíPaid / Void` (Overdue derived);
SO `Draft‚ÜíConfirmed‚ÜíClosed / Cancelled`; Bill `Draft‚ÜíOpen‚ÜíPartially Paid‚ÜíPaid / Void`.

## 5. Accounting ledger linkage (Books ‚Äî later)

The GL is a **projection of documents**, added later, joined by the **same
back-reference pattern the stock ledger uses**.

Three tables:
- `accounts` ‚Äî chart of accounts (`code, name, type(asset|liability|equity|income|expense), parent_id`). Seeded: AR, AP, Sales, COGS, Inventory Asset, Sales Tax Payable, Input Tax, Bank, Cash.
- `journal_entries` ‚Äî GL header: `date, description, source_type, source_id, is_posted`. **`(source_type, source_id)` is the link** back to the document/payment.
- `journal_lines` ‚Äî `journal_entry_id, account_id, debit, credit, party_id`. Must balance.

Posting map (each finalize emits one balanced entry):
```
Invoice:                       Payment received:
  Dr AR              total        Dr Bank/Cash      amount
    Cr Sales         subtotal       Cr AR           amount
    Cr Sales Tax     tax
  (perpetual:)                 Bill:
  Dr COGS            cost         Dr Inventory/Exp  subtotal
    Cr Inventory     cost         Dr Input Tax      tax
                                    Cr AP           total
```

Clip-on mechanism: a `PostingService.post(document)` runs on finalize **in the
same transaction** (mirrors how we upsert `stock_levels`). Account choices live
in a configurable account-mapping setting. Because every document is a complete
source record, the GL can be **backfilled** from existing documents at launch ‚Äî
no data migration. Operational AR/AP (on documents) works without the GL.

### 5.1 Accounting-readiness hooks (maintained from Phase 1)

Each phase must preserve these so Books connects later with zero rework:

- **Documents** store the full monetary breakdown (subtotal, discount, tax,
  shipping, total) and per-line amount + tax ‚Äî the exact numbers a journal entry
  debits/credits.
- **Payments** store amount, method, date, party, and allocations to documents ‚Äî
  enough to post `Dr Bank / Cr AR` (or the reverse for Made).
- **Stock movements carry a unit cost** (valuation input). Perpetual COGS +
  Inventory-Asset postings need cost-of-goods on each stock-OUT; recording cost
  on movements now (average/FIFO) means the GL can value them later without
  reconstructing history. *(Cost capture can be added when stock coupling lands;
  flagged here so it isn't forgotten.)*
- **Products / categories** are the anchor for later account mapping (income /
  COGS / inventory accounts) ‚Äî no accounting fields needed now, just stable ids.
- **Tax** is explicit and per-line (`tax_rates` + line `tax_amount`) so tax
  liability posts cleanly.
- **No hard deletes after finalize.** Corrections are new documents (credit /
  debit notes), preserving an auditable source trail the GL can rely on.
- **The back-reference is universal**: every document *and* payment is
  addressable as `(source_type, source_id)` ‚Äî the single join the GL uses.

## 6. Build phases

| Phase | Scope |
|---|---|
| **1 ‚Äî Engine + Invoices** | `documents`/`document_lines`/`document_sequences`, `tax_rates` (seed 18% + exempt), DocumentService (create/edit/finalize/void, numbering, totals), flat `/products/sellable` picker, invoice list/form/view, **stock-OUT on finalize**, tests. *No FBR.* |
| **2 ‚Äî Payments Received** | `payments`+`payment_allocations`, apply-to-invoice, Paid/Partial + balance |
| **3 ‚Äî Sales fulfillment** | Sales Order (feeds *committed*) + **Delivery Challan** *(optional)* + Sales Receipt; SO‚ÜíChallan‚ÜíInvoice chain for those who dispatch-then-bill, invoice skips stock move when shipped via challan; cash-sale receipt. Invoice-only remains the default path (Phase 1). |
| **4 ‚Äî Purchases** | PO (feeds *incoming*) + **Goods Receipt** (stock-IN) + Bills + Payments Made; PO‚ÜíGRN‚ÜíBill chain |
| **5 ‚Äî Returns / Credit Notes** | reverse-direction docs |
| **6 ‚Äî FBR e-Invoicing (last)** | product tax fields (HS/UoM/sale-type/SRO) + **per-item tax rate**, party province+registration, org province+FBR settings+token, FBR reference-data cache + sync, `FbrClient` (validate/post, sandbox), line tax-breakdown fields, **conditional posting on finalize** (IRN/QR/retry), debit/credit notes, scenario testing |
| **7 ‚Äî Accounting GL (Books)** | separate product; accounts, journal entries/lines, PostingService hook, backfill, P&L / Balance Sheet / AR-AP aging |

**FBR is the final phase** of the invoicing product (per request). Core commerce
(1‚Äì5) ships first with a simple line tax rate; the FBR fields, per-item tax rate,
and posting are added last ‚Äî just another autogenerated migration when we get
there. Accounting GL (Books) is a separate product after that.

## 7. Open decisions (confirm before Phase 1)

1. Shared `documents` engine (one table + `type`) ‚Äî **recommended**.
2. Stock leaves on **dispatch**. The **Delivery Challan is optional** ‚Äî a direct
   Invoice does the stock-OUT itself (the default, Phase 1). For businesses that
   dispatch-then-bill, an optional Challan does stock-OUT and the Invoice bills
   only. Challan built in Phase 3; nothing forces its use.
3. Default standard rate **18%** (current FBR standard) ‚Äî confirm.
4. Start at **Phase 1 (Invoices)**. FBR (incl. per-item tax rate) is Phase 6, last.
